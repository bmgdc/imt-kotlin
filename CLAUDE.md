# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this
repository.

## What this repository is

A Kotlin port of the IMT calculator (Portuguese property transfer tax, plus stamp duty)
from Bruno's `realtor_blueprint` project, built as a documented Claude Code exercise
attached to a job application. **The process is part of the product**: prompts, DEVLOG,
and git history are deliverables, not by-products. The repo is public — write every
commit, comment, and log entry as if the hiring team is reading it.

## Provenance and parity rules

- `oracle/` contains the reference JS implementation — Bruno's own prior work, vendored
  read-only. **Never edit the files in `oracle/`** except `generate-golden.mjs`. The
  original `*.test.js` files are reference material only; they are never executed here.
- `frontend-reference/` contains the original React frontend and its strings, also
  Bruno's own prior work, vendored read-only. Its `NOTES.md` defines what the static
  frontend must emulate, what to strip, and the server-side-calculation twist.
- `oracle/golden.json` is generated by `node oracle/generate-golden.mjs`. **Never
  hand-edit it.** If a parity test fails, the Kotlin is wrong — fix the Kotlin. Editing
  golden data to make tests pass defeats the entire project.
- Behavioral deviations from the JS are forbidden until full parity is reached. After
  that, improvements are allowed only behind new tests and a DEVLOG entry explaining the
  deviation.
- **When fixing parity failures** (inert until `ParityTest` exists): track the
  failing-case count across attempts. If a fix doesn't reduce it, or this is the second
  consecutive attempt diagnosing the same root cause, stop — don't try another fix.
  Append a DEVLOG entry titled "Parity loop stuck" (what's failing, what was tried, why
  it didn't work, what a human should check), commit, and end the session there instead
  of grinding on: repeated attempts against a stuck state risk rationalizing a wrong fix
  as done.
- The domain logic is legally sourced (CIMT Art. 17.º and related). The JS comments cite
  the articles; carry those citations into KDoc when porting. Do not reinterpret the law —
  parity with the oracle is the only correctness standard here.

## Commands

```
./gradlew test                     # unit + parity tests
./gradlew run                      # Ktor server on :8080 (blocking — Ctrl+C to stop)
node oracle/generate-golden.mjs    # regenerate golden.json (rarely needed)
```

`./gradlew run` blocks in the foreground serving requests; it will not reach 100% or
return until stopped. Verify against `http://localhost:8080`, not whatever URL an
editor's port-forwarding panel suggests — those can point at the wrong port.

CI (GitHub Actions) runs `./gradlew test` on every push. A red build never gets a feature
commit on top; fix it first.

## Architecture

- `src/main/kotlin/imt/` — pure domain, no framework imports:
  - `Tables.kt` — raw bracket data per year, enrichment ("parcela a abater" deductions),
    islands scaling (×1.25 on hpp/secondary/jovem limits and the stamp-duty threshold)
  - `TaxCalculator.kt` — accumulated-tax progressive computation (`computeTax`)
  - `ImtEngine.kt` — the engine: per-buyer precedence chain
    (offshore 10% > non-resident flat rate > urban-rehab exemption > post-rehab first
    transfer (hpp only) > jovem > standard), stamp duty with the jovem discount,
    per-buyer breakdown, non-resident potential-refund estimate
  - `Rounding.kt` — currency rounding, mirroring the JS epsilon method
- `src/main/kotlin/api/` — Ktor routes and DTOs; `POST /api/imt` and
  `GET /api/tables/{year}/{location}` (enriched tables + `nonResidentRate`, for the
  frontend's reference tables and display rules)
- `frontend-reference/` — the original React frontend, read-only (see above)
- `src/main/resources/static/` — `index.html` + `app.js`, the entire frontend: a reduced
  emulation of `frontend-reference/` per its `NOTES.md`; vanilla JS, Tailwind Play CDN,
  no build step
- `src/test/kotlin/` — ported unit tests plus `ParityTest.kt`, which is driven by
  `oracle/golden.json`

## Non-negotiable technical conventions

- **Arithmetic**: `Double` end to end, mirroring the JS operation order line-for-line in
  `computeTax`, table enrichment, and rounding. Do not refactor arithmetic expressions
  "for clarity" — reordering float operations breaks parity. No BigDecimal.
- **Rounding**: JS `Math.round` rounds ties toward +∞. Use `Double.roundToLong()` (JVM
  `Math.round`), **not** `kotlin.math.round` (ties-to-even). All currency rounding goes
  through `Rounding.kt`.
- **Infinity**: `Double.POSITIVE_INFINITY` internally for unbounded brackets; `null` at
  every JSON boundary (golden.json, API DTOs). Convert only at the edges.
- **Naming**: the JS says `youngun`; this port says `jovem` (the legal term). Map the name
  at the oracle boundary when comparing against golden data.
- **Frontend**: emulate `frontend-reference/`, don't redesign it — same layout,
  interaction rules, and copy (`strings-en.jsx` verbatim, including the tooltips' legal
  content); strip exactly what its `NOTES.md` says to strip, add nothing.

## Process protocol

- Start every session by reading `DEVLOG.md` (latest entry) and, if mid-milestone, the
  relevant `prompts/NN-*.md`.
- The session's kickoff prompt is committed verbatim to `prompts/` before the work starts.
- Commit at natural checkpoints with short imperative messages ("Port computeTax",
  "Wire parity suite"). Claude commits directly in this repo; keep the
  `Co-Authored-By: Claude` trailer.
- End every session by appending a DEVLOG entry: date, goal, what happened, decisions
  made, surprises, next step.
